## Static vs Dynamic Pages

A **static page** is a one-to-one relationship: one Route, one Page, one Layout.

<DiagramStaticPage />

A **dynamic page** goes further: many routes each point to a different `PageData` record, all rendering through one shared Page template. This is how a blog works — one template, infinite articles.

<DiagramDynamicPage />

One `BlogDetail` template. Every blog article gets its own URL, its own title and meta tags, its own content — all driven by data, not separate pages.

## Creating a PageData Entity

Extend `AbstractPageData`. It inherits from `AbstractPage`, which provides `title` and `metaDescription` used by the SEO system:

```php
// src/Entity/BlogArticleData.php
namespace App\Entity;

use ApiPlatform\Metadata\ApiFilter;
use ApiPlatform\Metadata\ApiResource;
use ApiPlatform\Filter\OrderFilter;
use ApiPlatform\Filter\SearchFilter;
use Doctrine\ORM\Mapping as ORM;
use Silverback\ApiComponentsBundle\Annotation as Silverback;
use Silverback\ApiComponentsBundle\Entity\Core\AbstractPageData;

#[ORM\Entity]
#[ApiResource(
    mercure: true,
    order: ['createdAt' => 'DESC'],
    paginationItemsPerPage: 12
)]
#[ApiFilter(SearchFilter::class, properties: ['title' => 'ipartial'])]
#[ApiFilter(OrderFilter::class, properties: ['title', 'createdAt'])]
class BlogArticleData extends AbstractPageData
{
    public ?string $headline = null;
    public ?string $body = null;
}
```

## What AbstractPageData Provides

| Property | Source | Purpose |
|----------|--------|---------|
| `title` | `AbstractPage` | Required. Overrides the page template's `<title>` for this record |
| `metaDescription` | `AbstractPage` | Per-record meta description |
| `page` | `AbstractPageData` | ManyToOne → the Page template (required) |
| `route` | `AbstractPage` | The Route pointing to this record |
| `parentPage` | `AbstractPage` | Optional. The parent Page this record lives under (for nested hierarchies) |
| `parentPageData` | `AbstractPage` | Optional. The parent PageData this record lives under |

`title` is required (`nullable: false` is set via `#[AttributeOverride]` in `AbstractPageData`).

## URL Prefix Convention

All PageData endpoints are prefixed with `/page_data/`:

```
GET    /page_data/blog_articles        # collection (with pagination, filtering)
POST   /page_data/blog_articles        # create a new article
GET    /page_data/blog_articles/{id}   # single article
PATCH  /page_data/blog_articles/{id}   # update
DELETE /page_data/blog_articles/{id}   # delete
```

## Setting Up in the Admin

1. **Create the Page template** — in `/_cwa/pages`, create a Page with `isTemplate: true`, set `uiComponent` to your Vue component name (e.g. `BlogDetail`), and assign a Layout
2. **Create PageData records** — in `/_cwa/data/BlogArticleData`, create articles. Each article has a `title` (required for SEO) and your custom fields
3. Routes are generated automatically from the article title slug when no explicit route is set

## Component Positions from PageData

You can link a component group position to a field on the PageData record. This is how a page template's "hero image" slot gets its data from `blogArticle.image`:

In your fixture:
```php
$cwa->page('blog-detail', 'BlogDetail', 'main', null, null, true)
    ->group('content', function (GroupBuilder $group) {
        $group->pageDataPosition('htmlContent');  // links to BlogArticleData::$htmlContent
        $group->pageDataPosition('image');         // links to BlogArticleData::$image
    });
```

The `pageDataPosition` method creates a `ComponentPosition` that is resolved at render time from the matching field on the current PageData record.

## Combining with Publishable

Add `#[Silverback\Publishable]` to your PageData class to give articles a draft/publish workflow. Draft articles are only visible to admins:

```php
#[Silverback\Publishable]
#[Silverback\Timestamped]
#[ORM\Entity]
#[ApiResource(mercure: true, order: ['createdAt' => 'DESC'])]
class BlogArticleData extends AbstractPageData
{
    use PublishableTrait;
    use TimestampedTrait;

    public ?string $headline = null;
}
```

## Accessing PageData on the Front-End

In your page template Vue component:

```ts
const cwa = useCwa()

// The current page data resource (populated by middleware)
const pageData = computed(() => cwa.resources.pageData.value)
const pageDataIri = computed(() => cwa.resources.pageDataIri.value)

// Access your custom fields
const headline = computed(() => pageData.value?.data?.headline)
```

`pageData.value?.data?.title` and `pageData.value?.data?.metaDescription` are automatically applied as SEO meta tags by the module middleware.

## Route Generation

Routes for PageData records are auto-generated from the `title` slug unless you specify a route explicitly. A blog article titled "My First Post" gets the route `/my-first-post` (or `/parent-path/my-first-post` when nested).

Set a custom route in the fixture:
```php
$cwa->pageData($article, 'blog-detail', '/blog/custom-path', 'blog_article_custom')
```

## Nested Pages

Some content naturally lives at more than one level — an event site might have `/events`, `/events/2024`, and `/events/2024/conference`. CWA handles this by linking pages and page data into a parent-child hierarchy, with child route paths automatically prefixed from the parent.

<DiagramNestedPages />

`<CwaPage />` is a built-in component you place inside a page template to render the next depth level. The module wires up the correct child resources automatically.

### Setting `parentPage` and `parentPageData`

Set `parentPage` on a Page entity (or `parentPageData` on a PageData entity) to make it a child. The parent can be either:

- A **Page** — for static sub-pages (e.g. `/about/team` as a child of the `/about` Page)
- A **PageData** — for dynamic sub-pages (e.g. `/events/2024/conference` as a child of the `EventYear 2024` PageData)

In a fixture:

```php
// Parent: the EventList page
$eventListPage = $cwa->page('event-list', 'EventList', 'main', '/events');

// Child: EventYear pages nested under it
$cwa->page('event-year-2024', 'EventYear', 'main')
    ->nested(function (CwaFixtureBuilder $cwa) use ($eventListPage) {
        // Route is auto-generated as /events/2024 from title "2024"
    });
```

Or set `parentPage` directly on the entity:

```php
$childPage->setParentPage($parentPage);
// or for PageData children:
$childPageData->setParentPage($parentPage);
```

### Route Auto-Prefixing

Child routes are automatically prefixed with the parent's route path. A child page titled "2024" under `/events` gets the route `/events/2024`. A grandchild titled "Conference" gets `/events/2024/conference`.

### Cascading Route Changes

When you rename a parent route via `PATCH /_/routes/{id}`, pass `cascadeChildPaths: true` to update all child routes and create redirects from the old paths automatically:

```json
PATCH /_/routes/abc123
{
  "path": "/happenings",
  "cascadeChildPaths": true
}
```

The API will:
1. Update the parent route to `/happenings`
2. Update all child routes (e.g. `/events/2024` → `/happenings/2024`)
3. Create `301` redirects from every old path

### Fetching a Route's Children

Retrieve the full child tree for a route (admin access required):

```
GET /_/routes/{id}/children
```

Returns the complete hierarchy of child routes under the given route.

### The Resource Manifest

The API serves a depth-grouped resource manifest at `/_/resource_manifest/{id}` (works for both public route paths and admin UUIDs). The manifest is a `resource_iris` array of arrays — each inner array is one depth level:

```json
{
  "resource_iris": [
    ["/events", "/layouts/primary"],          // depth 0
    ["/events/2024", "/layouts/primary"],     // depth 1
    ["/page_data/event_years/456"]            // depth 2
  ]
}
```

The Nuxt module reads this automatically to know which pages and page data records to load at each level.
