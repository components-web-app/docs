COVER:
- Recommended pipeline structure (GitHub Actions):

    1. test.yml — runs on every push / PR:
        - PHP: composer install, php-cs-fixer, phpstan, phpunit
        - Nuxt: pnpm install, eslint, type-check, vitest
        - E2E: playwright tests against a Docker Compose test stack

    2. build.yml — runs on merge to main / tag push:
        - docker build --target runner --tag ghcr.io/org/app-php:${{ github.sha }}
        - docker push ghcr.io/org/app-php:${{ github.sha }}
        - Same for nuxt image

    3. deploy.yml — runs after build.yml succeeds:
        - helm upgrade --install cwa ./chart --set php.image.tag=${{ github.sha }} ...
        - Or: kubectl set image deployment/php php=ghcr.io/org/app-php:${{ github.sha }}

- Secrets in GitHub Actions:
    DATABASE_URL, JWT_PASSPHRASE, MERCURE_JWT_SECRET as GitHub repository secrets
    GHCR_TOKEN for pushing to GitHub Container Registry

- Example build job (PHP):
    - uses: docker/setup-buildx-action@v3
    - uses: docker/login-action@v3
      with: { registry: ghcr.io, username: ${{ github.actor }}, password: ${{ secrets.GITHUB_TOKEN }} }
    - uses: docker/build-push-action@v5
      with:
          context: .
          file: ./api/Dockerfile
          push: true
          tags: ghcr.io/${{ github.repository }}-php:${{ github.sha }}
          cache-from: type=gha
          cache-to: type=gha,mode=max

- Running migrations in CI/CD:
    — Before updating the Kubernetes deployment:
        kubectl run migrations --image=ghcr.io/.../php:$SHA \
            --env=DATABASE_URL=... \
            --restart=Never -- bin/console doctrine:migrations:migrate --no-interaction
        kubectl wait --for=condition=complete job/migrations --timeout=120s
    — Or use a Helm pre-upgrade hook (see Kubernetes page)

- Rollback strategy:
    — Keep the previous image tag in CI output
    — On failure: helm rollback cwa (reverts to previous release)
    — Database migrations: always write reversible migrations (Doctrine handles it)

- Environment-specific deployments:
    — Staging: deploy every merge to main, with a staging values.yaml
    — Production: deploy only on version tags (v1.2.3), require manual approval

- Playwright E2E in CI:
    — Spin up the full Docker Compose stack in the CI job
    — Run: npx playwright test
    — Upload artifacts: test-results/ (screenshots, traces on failure)
    — The CWA docs repo includes Playwright snapshots as a regression baseline
