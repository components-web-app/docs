        image: dunglas/mercure:latest
        env:
            MERCURE_JWT_SECRET: ""  # use secretRef instead

- Managing secrets (do not hardcode in values.yaml):
    — Create a Kubernetes Secret:
        kubectl create secret generic cwa-secrets \
            --from-literal=JWT_PASSPHRASE=... \
            --from-literal=MERCURE_JWT_SECRET=...
    — Reference in values.yaml via secretRef
    — Or use External Secrets Operator / Vault for production

- JWT keys as Kubernetes Secrets:
    kubectl create secret generic jwt-keys \
        --from-file=private.pem=config/jwt/private.pem \
        --from-file=public.pem=config/jwt/public.pem
    — Mount as a volume in the php pod

- Running migrations (one-time Job before rolling update):
    — Include a Kubernetes Job that runs bin/console doctrine:migrations:migrate
    — Use an init container or a Helm hook (helm.sh/hook: pre-upgrade)
    — Never run migrations in the container startup script (parallel pod starts = race condition)

- Ingress configuration:
    — api.yourdomain.com → php service
    — www.yourdomain.com → nuxt service
    — mercure.yourdomain.com → mercure service (or /hub path prefix on the API ingress)
    — TLS: cert-manager with Let's Encrypt

- Rolling updates:
    — PHP: RollingUpdate strategy; graceful shutdown via SIGTERM
    — Nuxt: RollingUpdate; old pods serve until new pods are Ready
    — Mercure: StatefulSet with single replica (or use a hosted hub for HA)

- Resource requests and limits (starting point):
    php:   requests: { cpu: 100m, memory: 256Mi }, limits: { memory: 512Mi }
    nuxt:  requests: { cpu: 50m, memory: 128Mi }, limits: { memory: 256Mi }

- Health checks:
    PHP: readinessProbe GET /healthcheck, livenessProbe GET /healthcheck
    Nuxt: readinessProbe GET /, livenessProbe GET /
