Every website has the same skeleton.

A **layout** — the persistent outer shell with header, navigation, and footer. **Pages** that live inside it, each arranging their own regions for content. And within those regions, the actual content **components**: headings, images, rich text, hero sections, navigation links.

This structure is so universal that CWA makes it a first-class data model. Layouts, pages, routes, and components aren't configuration files or routing tables — they're **records in a database**, created and managed through an API. That single shift changes what becomes possible:

- A content editor can create a page, restructure its regions, or publish a change **without a deployment or a developer**.
- Every open browser session sees that change **instantly** — no page reload, no polling. Powered by Mercure, wired up automatically.
- Components are small, independent API resources. Only what changes gets re-fetched and re-cached. Page loads stay fast as the site grows.
- Add one PHP annotation to a component type and it gets a **full draft/publish lifecycle** — role-based access, publish-time validation, the lot.

**Your job as a developer**: define what components look like and what data they hold. Everything else is handled.

## How it fits together

```
Layout              →  outer shell (header + <slot> + footer)
  Page              →  content structure inside the layout
    ComponentGroup  →  a named region ("hero", "main", "sidebar")
      Component     →  what you build: titles, images, text, links
```

You write one PHP entity, one Vue display component, and one admin tab per component type. The module resolves every URL to the right layout, page, and component tree — no routing code, no CMS wiring.

## The two parts

A CWA application has two sides that communicate through a typed REST API:

**The API** — a Symfony application with the API Components Bundle installed. You extend `AbstractComponent` to define content types, annotate them with `#[Publishable]`, `#[Uploadable]`, or `#[Timestamped]`, and run migrations. The bundle provides authentication, serialisation, Mercure real-time publishing, and all the admin endpoints.

**The Nuxt app** — a Nuxt application with the CWA module. You create layout files in `app/cwa/layouts/`, page templates in `app/cwa/pages/`, and component Vue files in `app/cwa/components/`. The module handles route resolution, resource fetching, SSR, Pinia store management, and the full inline CMS.

The mapping is automatic: a `Title` PHP class renders via `app/cwa/components/Title/Title.vue`. No manual registration.

## The template repository

The [components-web-app template](https://github.com/components-web-app/components-web-app) is the starting point for every CWA project. Clone it and you immediately have:

- Docker Compose stack (FrankenPHP, Nuxt, PostgreSQL, Mercure, static cache via Souin)
- JWT authentication pre-wired
- Example components, fixtures, and CI/CD scripts
- Helm chart for Kubernetes deployment

Clone it, load the fixtures, and you have a working site to build on top of.
