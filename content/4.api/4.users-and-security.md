COVER:
- The authentication system:
    - Cookie-based JWT (not Authorization header)
    - LexikJWTAuthenticationBundle manages tokens
    - Tokens stored in a secure HttpOnly cookie (api_components by default)
    - Token auto-refreshed by the API when expired — no explicit refresh calls from front-end
    - Refresh tokens stored internally (not exposed to front-end)

- AbstractUser: extend it for your User entity
    - Provided by the Flex recipe as App\Entity\User
    - Fields: email, username (kept in sync), password (hashed), roles, enabled, emailAddress (for email changes)
    - The repository looks up by email OR username
    - How to add custom fields: extend the class normally

- Full security.yaml configuration walkthrough:
    - role_hierarchy: ROLE_ADMIN: ROLE_USER
    - User provider: entity with class AbstractUser
    - Firewall: login (json_login), main (stateless + jwt guard)
    - logout path: /logout
    - access_control:
        - Anonymous access to /token/refresh, /password/reset, /password/update
        - Anonymous POST/PATCH to /component/forms (for contact forms etc.)
        - Authenticated required for all other mutations

- JWT key generation (the one-time setup):
    mkdir -p config/jwt
    openssl genpkey -out config/jwt/private.pem -aes256 -algorithm rsa -pkeyopt rsa_keygen_bits:4096
    openssl pkey -in config/jwt/private.pem -out config/jwt/public.pem -pubout
    — passphrase in .env.local as JWT_PASSPHRASE (never commit this)

- lexik_jwt_authentication configuration:
    set_cookies:
        api_components:
            lifetime: 604800  # 1 week
    — cookie name must match refresh_token.cookie_name

- Refresh token configuration:
    silverback_api_components:
        refresh_token:
            handler_id: silverback.api_components.refresh_token.storage.doctrine
            options:
                class: App\Entity\RefreshToken
            cookie_name: api_components
            ttl: 604800

- The user registration + verification flow:
    - POST /users to create a user (you define what fields to accept)
    - Email sent with verification link → /verify-email/[username]/[token]
    - Front-end page provided automatically by the Nuxt module

- Password reset flow:
    - POST /password/reset with email
    - Email sent with reset link → /reset-password/[username]/[token]
    - Front-end page provided automatically

- Email address change flow:
    - POST to new-email endpoint
    - Confirmation email sent to new address → /confirm-new-email/[username]/[newEmail]/[token]
    - Front-end page provided automatically

- Notification emails (Mailer):
    - MAILER_DSN environment variable
    - Email templates configurable in silverback_api_components YAML
    - Events you can listen to for custom email sending

- Switch user (impersonation):
    - Enabled in the security.yaml skeleton
    - Useful for support: switch_user: true
    - Only available to ROLE_ALLOWED_TO_SWITCH (add to ROLE_SUPER_ADMIN)
