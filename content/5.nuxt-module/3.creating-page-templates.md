---
title: Page Templates
description: How to create CWA page template components that define the content regions of a page using CwaComponentGroup and CwaPage.
---

A page template defines the structure of a page — the slots where admins add and manage content. The PHP `Page` entity's `uiComponent` field maps to your Vue file. For dynamic pages (`isTemplate: true`), many routes share one template but each route points to its own `PageData` record.

## File Convention

```
app/cwa/pages/Primary.vue    →  CwaPagePrimary
app/cwa/pages/BlogDetail.vue →  CwaPageBlogDetail
```

The file name (without `.vue`) must match the `uiComponent` value on the Page entity.

## Every Page Component Receives `:iri`

The module passes the page IRI as a prop. Declare it:

```ts
import type { IriProp } from '@cwa/nuxt/runtime/composables'
defineProps<IriProp>()
```

The `iri` prop is the IRI of the current Page entity. Pass it as the `location` to any `CwaComponentGroup`.

## Minimal Page Template

```vue
<!-- app/cwa/pages/Primary.vue -->
<template>
    <main>
        <CwaComponentGroup reference="content" :location="iri" />
    </main>
</template>

<script setup lang="ts">
import type { IriProp } from '@cwa/nuxt/runtime/composables'
defineProps<IriProp>()
</script>
```

## Multiple Content Regions

A real page usually has distinct zones — hero, main body, sidebar:

```vue
<template>
    <div :class="uiClassNames">
        <section class="hero">
            <CwaComponentGroup reference="hero" :location="iri" />
        </section>

        <div class="grid lg:grid-cols-3 gap-8">
            <div class="lg:col-span-2">
                <CwaComponentGroup reference="main" :location="iri" />
            </div>
            <aside>
                <CwaComponentGroup reference="sidebar" :location="iri" />
            </aside>
        </div>
    </div>
</template>

<script setup lang="ts">
import type { IriProp } from '@cwa/nuxt/runtime/composables'

const props = defineProps<IriProp>()
const cwa = useCwa()

const page = computed(() => cwa.resources.page.value)
const uiClassNames = computed(() => page.value?.data?.uiClassNames ?? '')
</script>
```

## Dynamic Pages: Accessing Page Data

When the page is a template (`isTemplate: true`), `cwa.resources.pageData` contains the current record's data:

```ts
const cwa = useCwa()

const pageData = computed(() => cwa.resources.pageData.value)
const headline = computed(() => pageData.value?.data?.headline)
const createdAt = computed(() => pageData.value?.data?.createdAt)
```

The module automatically sets the `<title>` and `<meta name="description">` from `pageData.data.title` and `pageData.data.metaDescription`. You don't need to call `useHead` for the basics.

## Registering Pages

```ts
// nuxt.config.ts
cwa: {
    pages: {
        Primary: {
            name: 'Standard Page',
            classes: [
                { value: '', label: 'Default' },
                { value: 'full-width', label: 'Full Width' }
            ]
        },
        BlogDetail: {
            name: 'Blog Article',
            classes: []
        }
    }
}
```

## Page Load State

```ts
const isPageLoading = computed(() => cwa.resources.isLoading.value)
const progress = computed(() => cwa.resources.pageLoadProgress.value)
// { total: 8, complete: 5, percent: 62.5, resources: [...IRIs] }
```

Use `progress.percent` to drive a loading bar if you need one.

## Nested Pages and `<CwaPage />`

When your page hierarchy has multiple levels — a listing page containing detail pages — you use `<CwaPage />` inside a template to render the child level.

<DiagramNestedPages />

`<CwaPage />` is a built-in component that reads the current routing depth and renders whatever Page or PageData the router resolved for that level. You don't pass it any props — it works automatically.

### Adding a Child Slot to a Template

Place `<CwaPage />` wherever you want child content to appear:

```vue
<!-- app/cwa/pages/EventList.vue -->
<template>
    <div>
        <header>
            <CwaComponentGroup reference="hero" :location="iri" />
        </header>

        <!-- Renders the depth-1 page (e.g. EventYear or EventDetail) -->
        <CwaPage />
    </div>
</template>

<script setup lang="ts">
import type { IriProp } from '@cwa/nuxt/runtime/composables'
defineProps<IriProp>()
</script>
```

When a visitor is at `/events`, the `<CwaPage />` slot renders nothing (no depth-1 page). When they navigate to `/events/2024`, `<CwaPage />` renders the `EventYear` page template for that year. Navigation between child pages is smooth — only the `<CwaPage />` subtree re-renders.

### Accessing PageData Inside a Template Component

When a page template is rendered at depth 1 or deeper, use the `cwa-page-data-iri` inject key to get the PageData IRI for the current depth. This is provided automatically by `<CwaPage />`:

```vue
<!-- app/cwa/pages/EventDetail.vue -->
<template>
    <article>
        <h1>{{ title }}</h1>
        <CwaComponentGroup reference="content" :location="iri" />
    </article>
</template>

<script setup lang="ts">
import { inject, computed } from 'vue'
import type { IriProp } from '@cwa/nuxt/runtime/composables'
import type { ComputedRef } from 'vue'

defineProps<IriProp>()
const cwa = useCwa()

// pageDataIri is provided by <CwaPage /> — it's the IRI of the PageData for this depth
const pageDataIri = inject<ComputedRef<string | undefined>>('cwa-page-data-iri')

const eventData = computed(() => {
    if (!pageDataIri?.value) return null
    return cwa.resources.getResource(pageDataIri.value).value
})

const title = computed(() => eventData.value?.data?.title ?? '')
const headline = computed(() => eventData.value?.data?.headline ?? '')
</script>
```

::callout{icon="i-heroicons-light-bulb"}
`cwa-page-data-iri` is a `ComputedRef<string | undefined>` — always check `pageDataIri?.value` before using it, since not all pages have associated PageData.
::

### Depth-Aware API Methods

When called without arguments from inside a template component, `pageIriAtDepth()` and `pageDataIriAtDepth()` automatically resolve to the correct depth using the `cwa-page-own-depth` inject key provided by `<CwaPage />`:

```ts
const cwa = useCwa()

// These automatically use the depth of the current rendering context:
const pageIri = cwa.resources.pageIriAtDepth()     // ComputedRef<string | undefined>
const pageDataIri = cwa.resources.pageDataIriAtDepth() // ComputedRef<string | undefined>

// Or pass an explicit depth:
const depth0Page = cwa.resources.pageAtDepth(0)    // ComputedRef<resource | undefined>
const depth1PageData = cwa.resources.pageDataAtDepth(1)

// How many depth levels are in the current manifest:
const depthCount = cwa.resources.depthCount        // ComputedRef<number>
```

The no-arg form is what you want inside template components — you don't need to know what depth you're at.

### Building a Breadcrumb or Tab Bar

Use `depthCount` and explicit depth access to build navigation that spans all levels:

```vue
<script setup lang="ts">
const cwa = useCwa()

const breadcrumbs = computed(() => {
    const count = cwa.resources.depthCount.value
    return Array.from({ length: count }, (_, depth) => {
        const page = cwa.resources.pageDataAtDepth(depth).value
            ?? cwa.resources.pageAtDepth(depth).value
        return {
            title: page?.data?.title ?? '',
            iri: cwa.resources.pageIriAtDepth(depth).value,
        }
    })
})
</script>
```

### `pagesDepth` Configuration

To tell the module how deep to expect page hierarchies, set `pagesDepth` in `nuxt.config.ts`:

```ts
cwa: {
    pagesDepth: 3  // supports up to 3 levels: depth 0, 1, 2
}
```

This controls how many levels the auto-provided page routes resolve.
