COVER:
- What happens without any code changes on your end:
    - An admin publishes a component
    - The API broadcasts a Mercure update to the hub
    - Every browser with the page open receives the event
    - The Nuxt module's resource store updates automatically
    - Vue's reactivity re-renders the component with the new content
    - A "content updated" toast notification appears to inform visitors

- No front-end code needed:
    — The module subscribes to the Mercure hub in its Nuxt plugin
    — Topics are derived automatically from the loaded resources' IRIs
    — You don't write any EventSource or WebSocket code

- Enabling Mercure on the PHP side (per entity):
    #[ApiResource(mercure: true)]
    class Title extends AbstractComponent { ... }
    — Without mercure: true, the resource updates silently (no live broadcast)

- Infrastructure requirements:
    - A Mercure hub running alongside your API
    - MERCURE_URL (internal API → hub address) and MERCURE_PUBLIC_URL (browser → hub address)
    - MERCURE_JWT_SECRET for the publisher JWT
    - Docker Compose template wires this up out of the box

- The notification behaviour:
    — The toast is shown to non-admin visitors (admins see their own edit reflected immediately)
    — Visitors can dismiss it; dismissing does not prevent the store from updating
    — The content update is reactive: dismissing or not, new data is already in the store

- Custom Mercure topics (advanced):
    — By default topics = resource IRI strings
    — For custom subscription topics: configure in the API bundle
    — Nuxt module subscribes to topics based on resources it has fetched

- Troubleshooting:
    — No live updates: check MERCURE_PUBLIC_URL is reachable from browsers (not internal hostname)
    — Verify mercure: true is on the entity
    — Check browser DevTools → EventSource connections on the network tab
    — Hub 401 errors: check the MERCURE_JWT_SECRET matches between API and hub
